TASK DEFINITION -- HEILBRONN TRIANGLE PROBLEM (ADVERSARIAL IMPROVER)

You are an IMPROVER (Discriminator) in a two-population adversarial co-evolution game, analogous to a GAN:
- YOUR OPPONENTS (Pop A, the "Generator"): evolve point configurations with high min_area sitting in deep local optima
- YOU (Pop B, the "Improver"): take their configuration and PROVE IT IS SUBOPTIMAL by returning one with strictly higher min_area

THE GAME
Your opponents are other evolving programs. They produce (11,2) point arrays and try to make them as hard to improve as possible. As the Generator population evolves, their configurations sit in progressively deeper local optima with tighter basins. A simple random jitter that worked early will stop finding gains — you must evolve increasingly sophisticated improvement strategies to keep scoring.

Each evaluation you are tested against the current top-K Generator Hall-of-Fame and also tracked over your lifetime. Your standing is determined by two first-class metrics — see the METRICS section of your prompt for exact formulas and ranges:

- `fitness` — tanh-smoothed mean improvement against the current K=3 Generator HoF. >0.5 means you improved the configs on average; 0.5 is neutral; <0.5 means you made them worse.
- `wins` — lifetime count of distinct Generators you have EVER strictly improved (delta > 0). Grows monotonically. Distinguishes a one-trick improver from a broadly-capable one.

HALL-OF-FAME OPPONENTS + WHITE-BOX ACCESS
The Generator configurations you face are the deterministic top-K of Pop A's Hall of Fame, re-fetched every evaluation (the K best Generators alive right now). The opponent set changes when Pop A's HoF rotates, not on every mutation — so your lineage insights and past metrics are RELEVANT when the HoF is stable and SHIFT discretely when a new G displaces an old one.

You ARE shown the source code of the SAME Constructor you are evaluated against this turn (see "TARGET CONSTRUCTOR SOURCE CODE" block, when present). The source you see has already survived competitive selection — it was scored against many Improvers and made it into the archive. Surface-level or naive improvements have already been tried and discarded; what remains is genuinely hard to improve.

Use the source to:
  - Reason about WHY the configuration is suboptimal in the first place
  - Identify the specific assumption the algorithm relies on
  - Craft an improvement that addresses that assumption
  - Keep the strategy GENERAL ENOUGH that it improves many different Generators over time — `wins` grows by beating NEW Generators, not by beating the same one repeatedly

WHAT WINS THIS GAME
- Strategies that generalize across Generator lineages (a broadly-applicable local optimizer beats a one-shot exploit)
- Strategies that can find improvements even on configurations that already resist naive jitter
- Strategies that remain competitive as the Generator population evolves toward deeper optima — flexibility beats specialization

OBJECTIVE
Write an `improve(points)` function that:
1) Takes an (11, 2) NumPy array of point coordinates (inside unit-area equilateral triangle)
2) Returns an (11, 2) NumPy array of improved coordinates
3) The improved configuration must satisfy ALL constraints (containment, distinctness, non-degeneracy)
4) The improved configuration should have HIGHER min_area than the input

CONSTRAINTS ON OUTPUT
- Enclosing triangle: equilateral, area = 1.0, flat-bottomed
  Vertices: A=(0,0), B=(1.5197,0), C=(0.7598,1.3161)
- Exactly 11 distinct points, all within or on boundary
- No collinear triplets
- If your improvement violates any constraint, it counts as zero improvement

HELPER FUNCTIONS (must be imported from helper.py — not available globally):
  from helper import get_unit_triangle, get_smallest_triangle_area, is_inside_triangle

- `get_unit_triangle() -> (A, B, C)`
    Returns 3 arrays each shape (2,) — vertices of the unit-area equilateral triangle.

- `get_smallest_triangle_area(coordinates: np.ndarray) -> float`
    Input:  (n, 2) float array of n points
    Output: minimum area among all C(n,3) triangles formed by the points

- `is_inside_triangle(points, a, b, c) -> bool`
    Input:  points — shape (n, 2) or (2,) for a single point; a, b, c — each shape (2,) vertices
    Output: True if ALL given points lie inside or on the triangle boundary

OUTPUT FORMAT
Implement `def entrypoint():` that returns a callable:
    `improve(points: np.ndarray) -> np.ndarray`
where points is (11, 2) and the return is (11, 2).

Use numpy, scipy, sklearn, or standard library as needed.
Fix random seeds if using randomness.
