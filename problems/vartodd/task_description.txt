Your global task is to mutate a Python tuner that minimizes T-count by tuning a FastTODD policy.
Objective: discover policies that reach the lowest decomposition rank (T-count) reliably across seeds.
Current SOTA: 397

---
## TODD overview
- Decomposition is represented by a matrix. Rank = number of columns = T-count.
- Each step: generate actions, then choose according to the policy for the current rank.
- Each action produces a new matrix with fewer rows (lower rank).
- Two stages:
  - TOHPE: restricted search space, easy to find good z
  - TODD: heavier exploration, higher time cost

---
## FastTODD policy variables
### 1) Search policy variables
- min_z_to_research: int — main complexity of action generation during TODD (how many z to check)
- beamsearch_width: int — keeps best matrices after each iteration according to final_weights; runtime scales ~linearly
- todd_width: int — number of actions per iteration (if 1 there is no beamsearch)
- gen_part: float in [0,1] — fraction of nullspace generators to check
- num_samples: int — samples per candidate z. If num_samples >= 2**bd, evaluate the full space
- min_pool_size: int — minimum number of actions found during exploration
- max_pool_size: int — number of best actions kept during exploration
- tohpe_num_best: int — how many z from TOHPE
- try_only_tohpe: bool — if True and TOHPE found > min_pool_size actions, skip TODD; otherwise run TODD
- max_tohpe: int — number of actions chosen during TOHPE
- max_from_single_ns: int — limit number of actions chosen from one z
- max_reduction, min_reduction: int — clamp allowed rank reduction; if not satisfied, push other actions
- temperature: float in [0,1] — softmax temperature for final choice

### 2) Choice policy variables
- pool_weights: ExplorationScore(weights=[wred, wdim, wbucket, wvw, wz], centers=[cred, cdim, cbucket, cvw, cz], pow=pow)
- final_weights: FinalizationScore(weights=[wred, wdim, wbucket, wvw, wz, wtohpe], centers=[cred, cdim, cbucket, cvw, cz, wtohpe], pow=pow)

Feature meanings:
- red: rank reduction from an action
- dim: dimension of space from which action was sampled (num_samples <= 2**dim)
- bucket: early estimate of reduction for a given z
- vw: how much the matrix changes for a given action
- z: number of ones in z boolean vector
- tohpe: size of future TOHPE space (can be very time-consuming if max_pool_size is big; wtohpe=0 speeds evals)
- pow: can be negative

Score formula:
score = sum_i w_i * |x_i - c_i|^pow, where x_i in [0,1]
Default vectors are zero; using all parameters can be too greedy.

## Rank scheduling
All parameters may be represented as:
- a constant value, or
- a rank schedule: two lists rank_thresholds (sorted high → low) and values.

Schedule semantics:
Thresholds are sorted descending by code before evaluation. The rule is:
for i in range(0, len(rank_thresholds)):
    if rank_thresholds[i] <= current_rank_in_todd:
        return values[i]
return values[-1]

Example:
rank_thresholds = [500, 450, 420], values = [A, B, C]
- rank=460 -> A
- rank=430 -> B
- rank=410 -> C

---
## Runtime notes
- beamsearch_width and todd_width scale runtime roughly linearly.
- num_samples can dominate runtime if it approaches 2**bd; keep it small unless you need exhaustive search.
- min_z_to_research and gen_part also scale candidate generation; increasing both can explode cost.
- max_pool_size increases evaluation cost; keep it as small as possible while preserving quality.
- If you must push exploration, do it in early ranks only via scheduling.

## Reusing backed-up paths
During initialization, choose path_name to load a path from previous evolutions.
- "init" is reserved for pure runs without preloaded paths
- When path_name != "init", you MUST also pass init_rank_thr to select the init matrix from the loaded path. init_rank_thr must be at least final_rank + 50
- This load is only used to pick the starting matrix and record loaded_rank; the run itself behaves like "init".
- Use other names you will see in evo path statistics
- prefer using "init" or higher threshold with known path

Example:
  fun = Evaluator(path_name="init", init_rank_thr=0)  # start from scratch (recommended default)
  fun = Evaluator(path_name="<name_from_evo_path_statistics>", init_rank_thr=450)  # reuse a saved path

How to pick path_name:
- Use names from "evo path statistics" in the latest report.
- Prefer lower ranks first; break ties by choosing the newest hash.
- Name format: "i<init_rank>_m<init_rank_thr>_f<final_rank>_<hash>".
- The keyword "init" is reserved for starting without preload.
- Choose init_rank_thr to control how deep you jump into the loaded path (lower = deeper).
- If stats are missing, the path may predate incoming-info saving; still valid but will show unavailable stats.

## Evaluator interface (must follow)
- Implement Evaluator.policy_mapping(), Evaluator.__call__(), and entrypoint() returning fun.get_best().
- evaluator.policy_mapping(): set parameters for TODD search
  - Use evaluator.map_par(mapping=<callable>, thr=<int>) to declare tunable parameters.
    Each call returns mapping(hidden_parameters[cur_idx]) and increments cur_idx
    only if init rank > thr; otherwise it returns zero and does not activate.
  - evaluator.set_X(constant) — set a constant schedule
  - evaluator.set_X(ranks: List[int], values: List[Any]) — set a rank schedule
- evaluator(params): evaluate loss for given parameters
  - evaluator.run(params, seeds) -> List[int] ranks per seed
- evaluator.best_paths: list of best paths found
- evaluator.best_rank: current best rank
- evaluator.path_num: number of best paths available for reinitialization
- evaluator.set_up_new_init(path_num, rank_thr, xopt) -> List[float] | None
  - selects last matrix with rank > rank_thr from a best path and updates init
  - returns None if no matrix exceeds rank_thr; else returns active parameters
- evaluator.extract_active(): return current active parameters

## Hints for path info
- bd: basis_dim for action's z (2**bd is max useful sampling)
- r: total rank reduction
- bs: bucket size
- m: mean, d: diff from best, q: quantile vs others
- tha: TOHPE actions added to pool
- tda: TODD actions added to pool
- 14/194/550: callback stats about reaching best reduction during optimization
- total_fasttodd_evals ≈ max_fun_evals * len(seeds) * beamsearch_width
  Example: 4 seeds, width=10, PSO swarmsize=50, maxiter=10 -> 4*10*50*10=20000 evals (too high; avoid)
- quantile statistics over all T-counts after FastTODD
- best_policy: dict with best policy parameters
- search_stat: optimization info with ranks reached at eval steps
- total_eval: total number of FastTODD evaluations
- best_seen_times: number of times final rank was reached
- evo path statistics names are the paths used for evaluator initialization:
  - i{init}: init rank used for the saved path
  - f{num}: num is final found rank in path

## Constraints on code
- Must include Evaluator.policy_mapping, Evaluator.__call__, and entrypoint() returning fun.get_best().
- Use self.map_par(fun=<callable>, rank_thr=<int>) to obtain tunable parameters.
- The least rank threshold must be below the expected final rank.
  Example: if final rank cannot be <405, then [430,415,405,0] wastes parameters vs [430,415,0].
- Check rank scheduling for wasted thresholds.
  Example: observed ranks 567→407 but scheduling (410,405,390,0) is effectively (410,405).
- Estimate the lowest possible rank based on current reached ranks.
- Whole program runtime must be <3000s. Faster is better. Optimal time is ~1800s.
- Seeds must be fixed before any sampling.
- If loaded_rank is present and final rank is not better, validation adds a penalty.

## Rules to analyze search data/metrics and mutate
1) The only source of truth is Metrics output and best path info.
2) Interpret best policy values to extract strategy and reuse good parameters.
3) Analyze path statistics. Example: big best_count can imply bad exploration; good diff+fitness suggests penalize that value.
4) Analyze fixed parameters in policy_mapping.
5) Account for the very stochastic, ill-behaved landscape.
6) Analyze how many optimization parameters are used and how good search is; introduce biases and heuristics.
7) Find evidence of which parameters matter, can be fixed, or need deeper analysis.
8) Analyze/exploit/mutate the search strategy.
9) Focus on best policy parameters only; do not use preferences without evidence.
   Example: if wred < 0 is consistently best, keep penalization.
10) Check scheduling logic.
11) Use set_up_new_init to exploit found paths (avoid wasting compute on high ranks).
12) Balance the number of programs with init and without init.
13) Use no more than 10-15 mapped parameters; choose carefully.
