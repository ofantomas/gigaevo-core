PROMPT META-OPTIMIZATION FOR VARTODD CODE EVOLUTION

You are evolving the mutation prompt used by an evolutionary algorithm to guide
LLM-driven code mutations of vartodd matrix decomposition tuners.

Each candidate is a Python module exporting:
    entrypoint() -> dict

The returned dict has keys:
    "system": the mutation system prompt template (str)
    "user": the mutation user prompt template (str)

────────────────────────────────────────────────────
CONTRACTS — REQUIREMENTS FOR EVOLVED PROMPTS
────────────────────────────────────────────────────

The evolved prompt (system + user) must satisfy ALL of these contracts.
Violating any contract causes the mutation pipeline to fail silently, wasting
compute and hurting this prompt's fitness score.

CONTRACT 1 — SYSTEM PROMPT PLACEHOLDERS:
The system prompt MUST contain these exact placeholders (filled at runtime):
    {task_description} — full vartodd task description (injected by framework)
    {metrics_description} — fitness metrics and their optimization directions
Without these, the mutation LLM receives no information about the downstream task.

CONTRACT 2 — USER PROMPT PLACEHOLDER:
The user prompt MUST contain this exact placeholder:
    {parent_blocks} — parent program code, insights, lineage data, execution stats
Without this, the mutation LLM has no parent code to mutate.

CONTRACT 3 — JSON OUTPUT FORMAT:
The mutation LLM's response is parsed as JSON. The prompt (typically the user
prompt) MUST instruct the mutation LLM to return a JSON object with these fields:
    "code": str — complete mutated Python program (REQUIRED — raw code, no markdown)
    "archetype": str — name of the mutation strategy used
    "justification": str — reasoning linking insights to changes
    "insights_used": list[str] — which insights were acted upon
The "code" field is extracted and used as the child program. If the response is
not valid JSON or lacks "code", the mutation fails entirely.

CONTRACT 4 — PYTHON DOUBLE-BRACE ESCAPING:
Since the prompt templates are Python strings processed with str.format(), any
literal curly braces in the prompt text must be escaped as {{ and }}. For example,
a JSON example block must use {{  }} not {  }. Unescaped braces cause KeyError
at runtime.

────────────────────────────────────────────────────
DOWNSTREAM TASK: VARTODD MATRIX DECOMPOSITION
────────────────────────────────────────────────────

The mutation prompt you are evolving guides an LLM to mutate Python tuner code
that minimizes T-count via FastTODD policy optimization. Understanding this
domain is essential to writing effective mutation prompts.

WHAT THE EVOLVED CODE DOES:
Each program defines an Evaluator class with:
  - policy_mapping(): configures TODD search parameters using map_par() for
    tunable params optimized by PSO/CMA-ES
  - __call__(params): evaluates a parameter vector, returns loss
  - entrypoint(): runs the optimizer and returns fun.get_best()

KEY DOMAIN CONCEPTS:

1. TODD SEARCH POLICY — controls how the decomposition tree is explored:
   - beamsearch_width, todd_width: breadth of search (runtime scales linearly)
   - min_z_to_research: depth of action generation
   - num_samples: samples per candidate (if >= 2^bd, exhaustive)
   - gen_part: fraction of nullspace generators to check
   - temperature: softmax temperature for final action choice
   - max_pool_size, min_pool_size: action pool bounds
   - try_only_tohpe: if True, skip expensive TODD when TOHPE suffices
   - max_reduction, min_reduction: clamp allowed rank reduction per step

2. SCORING FUNCTIONS — how actions are ranked:
   - pool_weights (ExplorationScore): weights for [red, dim, bucket, vw, z]
   - final_weights (FinalizationScore): weights for [red, dim, bucket, vw, z, tohpe]
   - These control which decomposition actions are preferred at each rank

3. RANK SCHEDULING — different parameters at different matrix ranks:
   - Parameters can be constants or rank-scheduled: [(rank_thr, value), ...]
   - Critical for adapting strategy as matrix shrinks (high rank = easy, low rank = hard)

4. PATH REUSE — exploiting previous decomposition paths:
   - path_name selects a saved path from prior evolutions
   - init_rank_thr picks the starting matrix from that path
   - Loaded paths provide warm-starts; penalty if no improvement over loaded_rank
   - "init" keyword = fresh start without path reuse

5. OPTIMIZER STRATEGY — how policy parameters are searched:
   - PSO (pyswarms): good for exploration, particle swarm
   - CMA-ES (cma): good for exploitation, covariance matrix adaptation
   - Hybrid: PSO for exploration → CMA for refinement
   - Budget = num_eval × particles/popsize × seeds × beamsearch_width

FITNESS = rank + sum(P)/size(P), where P is the found decomposition matrix.
Lower is better. SOTA ≈ 397 on gf2^16. Sentinel = 420.

WHAT MAKES A GOOD MUTATION PROMPT:
- Guides the LLM to analyze the parent's best_policy, search_stat, and path info
- Encourages exploiting path reuse (init from known-good paths at strategic ranks)
- Steers rank scheduling to focus compute on the hard low-rank bottleneck
- Balances runtime budget: deeper search (more evals) vs broader search (more params)
- Uses evidence from evo path statistics to pick promising path_name values
- Avoids wasting map_par slots (max 10-15 useful); each adds an optimization dimension

────────────────────────────────────────────────────
FITNESS DEFINITION
────────────────────────────────────────────────────

A "successful" mutation = child program achieves strictly lower fitness (rank)
than parent. Prompt fitness = success_rate = fraction of mutations that achieved
strict improvement, aggregated across 3 independent evolution runs with
Bayesian Beta(1,3) prior.

The behavior descriptor is prompt_length (number of characters).
The primary fitness metric is success_rate.
