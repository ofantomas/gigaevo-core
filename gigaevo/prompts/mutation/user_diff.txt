EVOLUTIONARY MUTATION (DIFF MODE): Adaptive Code Evolution

Transform the program using program insights and historical lineage intelligence with intelligent exploration/exploitation balance. Emit your transformation as one or more **SEARCH/REPLACE blocks** against Parent 1, not as full source.

## INTELLIGENCE INPUTS

**PROGRAM INSIGHTS**: category [tag] (severity): concrete evidence
- Categories: LLM-generated 1-2 word labels
- Tags: beneficial, harmful, neutral, fragile, rigid
- Severity: high, medium, low

**TAG MEANINGS (evolutionary guidance)**:
- **beneficial**: Current pattern is good → PRESERVE/EXTEND this approach
- **harmful**: Current pattern is bad → REMOVE/AVOID this approach
- **fragile**: Current pattern is risky → IMPROVE/ROBUSTIFY this approach
- **rigid**: Current pattern is inflexible → MAKE MORE ADAPTABLE
- **neutral**: Current pattern has no clear impact → IGNORE for evolution

**LINEAGE INSIGHTS**: Historical mutation outcomes and their measured effects
- strategy: imitation/generalization/avoidance/exploration/refinement (past action taken)
- description: causal explanation with quantified impact (≤50 words)
- delta: measured performance impact (relative change)

**INSPIRATION TRANSITIONS**: Historical parent→child mutations sampled from other MAP-Elites elites

You may receive Inspiration Transition cards labeled `IT-1`, `IT-2`, etc. Each card describes a donor transition observed elsewhere in the population.

A card may contain:
- donor parent and donor child IDs
- donor parent fitness, donor child fitness, and measured delta
- supporting metrics such as mean_rel_energy, validity, and convergence
- mutation_output from the donor child when available
- lineage interpretation explaining why the transition may have helped or hurt
- diff hunks showing what changed in that donor transition

How to use these cards:
- Treat each card as evidence about an adaptation pattern, not as an additional parent.
- Use the adaptations as inspiration for mutating Parent 1.
- Prefer transferable mechanisms over donor-specific constants, names, or structure.
- The measured delta belongs to the whole donor transition; do not claim that each individual hunk caused that delta.
- Diff hunks in inspiration cards are historical evidence only. Do not copy them as SEARCH blocks unless the same text exists exactly once in Parent 1.
- All SEARCH/REPLACE blocks you emit must target Parent 1 only.
- If a card does not fit Parent 1's structure, ignore it.
- If multiple cards suggest compatible mechanisms, you may synthesize them into one coherent mutation.

Tracking inspiration use:
- If an inspiration card influenced your mutation, list only its card ID in `inspirations_used`, e.g. `["IT-2"]`.
- Do not list cards that you ignored.
- In `justification`, briefly explain the mechanism adapted from each used card.

**EVOLUTIONARY STATISTICS**: Population-level context
- Generation history table: fitness trends across generations (← marks current)
- Is useful to gauge population progress and inform exploration/exploitation balance

## EVOLUTIONARY ARCHETYPE SELECTION

Choose your evolutionary approach based on evidence strength and risk tolerance:

### EXPLOITATION ARCHETYPES (Evidence-Driven Refinement)
**When lineage shows consistent positive outcomes and strong beneficial insights:**

1. **Precision Optimization**
   → Fine-tune proven patterns, minimal risk changes, conservative improvements

2. **Proven Pattern Extension**
   → Replicate successful strategies, generalize working approaches, safe adaptations

3. **Harmful Pattern Removal**
   → Eliminate documented failure modes, clean up problematic code, defensive improvements

### EXPLORATION ARCHETYPES (Innovation-Driven Change)
**When lineage shows failures, weak evidence, or strong harmful/rigid insights:**

4. **Computational Reinvention**
   → Novel algorithmic paradigms, alternative data representations, constraint-driven design

5. **Solution Space Exploration**
   → New problem-solving strategies, unexplored search spaces, alternative optimization flows

6. **Approach Synthesis**
   → Combine multiple techniques, hybrid architectures, emergent computational behaviors

### HYBRID ARCHETYPES (Balanced Approach)
**When evidence is mixed or moderate confidence:**

7. **Guided Innovation**
   → Preserve proven elements while introducing targeted improvements

8. **Conservative Exploration**
   → Explore within safe boundaries, maintain structural integrity

## ARCHETYPE SELECTION FRAMEWORK

**Evidence Assessment**:
- **Strong positive lineage** + **beneficial insights** → Choose Exploitation archetype
- **Weak/negative lineage** + **harmful/rigid insights** → Choose Exploration archetype
- **Mixed evidence** + **moderate confidence** → Choose Hybrid archetype

**Risk Tolerance**:
- **High confidence** → Exploitation archetypes (1-3)
- **Low confidence** → Exploration archetypes (4-6)
- **Moderate confidence** → Hybrid archetypes (7-8)

## EXECUTION PRINCIPLES

- **Evidence-driven**: 1-3 program insights from distinct categories required
- **Archetype-appropriate**: Match change scope to selected archetype
- **Historical guidance**: leverage lineage outcomes while allowing for innovation
- **Traceability**: link all changes to specific insights or lineage evidence
- **Prioritization**: severity first, then beneficial/fragile tags preferred
- **Delta preference**: favor strategies with positive historical outcomes
- **Tag-based action**: Use harmful insights to REMOVE patterns, beneficial/fragile to PRESERVE/IMPROVE patterns
- **Archetype constraints**:
  - Exploitation: ≤2 small, localized changes
  - Exploration: Meaningful structural novelty required
  - Hybrid: Balance proven elements with targeted innovations

## LIGHTWEIGHT ADDITIONS (ADAPTIVE)
- **Plateau trigger:** If recent lineage shows small |Δfitness| magnitudes, prefer **Exploration** for this mutation.
- **Category diversification:** If a category recently produced harmful/negative Δfitness, explore a **different** category this time.
- **No-repeat harmful (unguarded):** Do **not** reintroduce a strategy tagged **harmful** in lineage unless accompanied by a clear **guard** or corrective mechanism.

## SEARCH/REPLACE FORMAT (STRICT)

Emit your change as one or more SEARCH/REPLACE blocks in the `code` field. The blocks are applied in order to Parent 1's source via literal string replacement.

**Each block has exactly this shape:**

```
<<<<<<< SEARCH
<verbatim block from Parent 1, copied byte-for-byte including indentation>
=======
<the replacement block (or empty to delete the SEARCH block)>
>>>>>>> REPLACE
```

**Rules (these are enforced — violations cause the mutation to be discarded):**

- **Verbatim SEARCH**: The text between `<<<<<<< SEARCH` and `=======` must appear EXACTLY ONCE in Parent 1's current source. Copy whitespace, indentation, blank lines, and trailing characters byte-for-byte. Do NOT paraphrase, reformat, or strip whitespace.
- **Uniqueness**: If the snippet you want to change appears in several places in Parent 1, expand SEARCH upward/downward to include enough surrounding lines that the match becomes unique. A SEARCH that matches zero times or more than one time is rejected.
- **Markers literal**: The markers `<<<<<<< SEARCH`, `=======`, and `>>>>>>> REPLACE` must each appear on their own line, exactly as written (seven `<`, seven `=`, seven `>`).
- **Multiple edits**: For multiple changes, emit multiple consecutive blocks. They are applied in order; each operates on the result of the previous one.
- **Replace whole logical units when possible** (a full function, a full if-branch, a full block) rather than a single line — easier to keep indentation consistent.
- **No outer wrapping**: The `code` field's contents start directly with `<<<<<<< SEARCH`. Do NOT wrap the blocks in ```markdown``` fences. Do NOT add commentary between blocks.
- **No full source**: Do NOT emit the entire mutated program. Only emit the SEARCH/REPLACE blocks.

**Tiny example.** Parent 1 contains:

```python
def score(x):
    return x ** 2

def main():
    print(score(3))
```

Acceptable `code` field to change the exponent to 3:

```
<<<<<<< SEARCH
def score(x):
    return x ** 2
=======
def score(x):
    return x ** 3
>>>>>>> REPLACE
```
