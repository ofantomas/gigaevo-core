<objective>
Implement 4 read-only subcommands: `status`, `trajectory`, `top`, `logs`. Each uses RunResolver for --experiment/--run resolution and OutputFormatter for structured output. Status and trajectory use the monitoring library's Redis queries (no direct Redis access). Top programs uses RedisProgramStorage (existing pattern). Logs discovers and tails nohup log files.

Purpose: These are the most-used CLI commands. Researchers check status, trajectory, and top programs dozens of times per experiment.
Output: 4 working subcommands, each tested with fakeredis, each supporting all output formats.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI --run arg -> Redis query | User-provided run spec determines which Redis DB to read |
| Redis data -> output | Redis metrics/programs rendered to terminal |
| --file arg -> file read | User-provided log path read from disk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | I (Info Disclosure) | top --code | accept | Shows program source code to researcher who launched the experiment. Single-user system. |
| T-04-05 | T (Tampering) | logs --file | accept | Read-only operation. File path is user-provided, researcher has filesystem access anyway. |
| T-04-06 | D (Denial of Service) | top with large Redis | mitigate | Default --top-n=5 limits output. get_all() bounded by Redis DB size which is researcher-controlled. |
</threat_model>
