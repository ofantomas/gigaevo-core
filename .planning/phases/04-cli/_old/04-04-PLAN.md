Purpose: Flush is the safety-critical cleanup command. Watchdog is how monitoring gets started for an experiment. Both must be CLI-accessible for the unified entry point.
Output: Working flush (with safety gates) and watchdog subcommands, tested.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| --confirm flag -> destructive Redis ops | Safety gate prevents accidental data loss |
| --db args -> Redis FLUSHDB | User specifies which DBs to destroy |
| experiment.yaml -> WatchdogEngine config | Manifest controls monitoring behavior |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-10 | T (Tampering) | flush without --confirm | mitigate | Dry-run by default. --confirm flag required. DB range validated 0-15. warn_if_not_archived() warns before flushing non-empty DBs. |
| T-04-11 | D (Denial of Service) | flush kills wrong workers | mitigate | Worker killing is scoped to target DBs only (find_exec_runner_pids filters by parent run.py PID). Same safety as tools/flush.py. |
| T-04-12 | E (Elevation) | watchdog --plugin | accept | Plugin override is a convenience flag for the researcher. No privilege boundary crossed. |
</threat_model>
