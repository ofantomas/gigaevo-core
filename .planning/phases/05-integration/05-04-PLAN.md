For each skill file, perform this audit:

1. **Find every `gigaevo` invocation** in bash code blocks.
2. **Verify the subcommand exists** in `_LAZY_SUBCOMMANDS`: analyze, checkpoint, closeout, collect, export, flush, launch, logs, manifest, plot, restart, status, top, trajectory, watchdog.
3. **Verify flag positions**: Global flags (`-e`, `-r`, `-f`, `-q`, `-v`, `--redis-host`, `--redis-port`) go BEFORE the subcommand. Subcommand-specific flags go AFTER the subcommand.
4. **Verify manifest subcommand syntax**: The correct pattern is `gigaevo -e "$EXP" manifest {get|set|update|gate|pr-description} [args]`.
5. **Verify `--format` placement**: `--format` / `-f` is a GLOBAL flag on the parent group. Correct: `gigaevo -e "$EXP" -f json manifest get runs`. If any skill puts `--format` AFTER `manifest`, move it before `manifest`.
6. **Check for non-existent commands**: Look for `gigaevo watch`, `gigaevo run`, or any other commands not in the registry.
7. **Check for `PYTHONPATH=.` workarounds**: Any remaining `PYTHONPATH=. python -c "from tools..."` should be replaced with the equivalent `gigaevo` CLI command.

**Specific known issue patterns to fix:**

Pattern A: `--format` after manifest subcommand:
- WRONG: `gigaevo -e "$EXP" manifest get runs --format json`
- RIGHT: `gigaevo -e "$EXP" -f json manifest get runs`

Check if this is actually an issue by reading manifest_cmd.py. If "get" subcommand has its own `--format` flag, the skill may be correct. If `--format` only exists on the parent group, all skills using `--format json` after `manifest` need fixing.

IMPORTANT: Read `gigaevo/cli/manifest_cmd.py` carefully to determine if `manifest get` or `manifest set` have their own `--format` flag. If not, `--format` must go before `manifest` on the CLI.

Pattern B: Missing `$EXP` variable:
- WRONG: `gigaevo manifest get status`
- RIGHT: `gigaevo -e "$EXP" manifest get status`

Pattern C: Wrong field names for Pydantic shape:
This is a SKILL file issue, not a CLI issue. The manifest CLI (`manifest get`) navigates fields by dot notation. With Pydantic manifest:
- `manifest get name` -> `manifest get experiment.name`
- `manifest get status` -> works if `_navigate_manifest` falls through to `experiment.status`
- `manifest get task` -> `manifest get experiment.task`

Check how `_navigate_manifest` in manifest_cmd.py handles these. If it searches nested objects, `manifest get status` may still work. If it only checks top-level, it needs `manifest get experiment.status`. READ manifest_cmd.py `_navigate_manifest` function to determine this.

If the function auto-searches nested fields (tries `getattr(manifest, field)` then `getattr(manifest.experiment, field)`), then `manifest get status` works. If not, update skills to use the full dotted path.

**Do NOT change any non-CLI content in skills** (experiment design methodology, analysis steps, etc.). Only fix CLI command invocations.
